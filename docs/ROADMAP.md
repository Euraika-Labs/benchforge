# BenchForge Roadmap

BenchForge is becoming a real desktop benchmark tool for local and cloud LLMs. The app now has verified end-to-end local/cloud benchmark paths, but it should still be treated as a pre-1.0 engineering workbench rather than a public leaderboard. This roadmap defines the remaining work required to make it broadly trustworthy for day-to-day model selection.

## Definition of Real

A usable BenchForge release must let a user:

1. Add local targets: Hugging Face GGUF through llama.cpp, Ollama, LM Studio, vLLM, MLX, or any local OpenAI-compatible server.
2. Add cloud targets: OpenAI, Anthropic, Mistral, OpenRouter, Azure OpenAI, Google Gemini, and generic OpenAI-compatible APIs.
3. Store API keys securely in macOS Keychain, never in project files or exports.
4. Select benchmark packs, repetitions, warmups, and deterministic generation settings.
5. Run benchmarks without freezing the UI, with cancellation, retry, and resumable history.
6. Compare pass rate, score, latency, p95 latency, tokens, throughput, cost, HTTP status, retry count, retry delay, and normalized errors.
7. Inspect prompts, responses, raw provider JSON, logs, patches, diffs, scorer output, and run configuration.
8. Export a reproducible report that another engineer can verify.

## Current Reality

Working or partially working:

- Tauri + React desktop app builds, runs, and packages as a macOS DMG.
- SQLite stores targets, runs, artifacts, jobs, run groups, and reproducibility metadata.
- Hugging Face GGUF search/download/start exists for llama.cpp and can create a normal local target.
- Cloud target setup exists for OpenAI-compatible APIs, OpenAI Responses, Anthropic, Mistral, OpenRouter, Azure OpenAI, and Google Gemini.
- Prompt packs (`llm-connectivity`, `llm-basics`, `llm-core`, `llm-practical`, `llm-decision-suite`, `llm-structured-output`, `llm-grounded-context`, `llm-reliability`) and quick-smoke code tasks can run and persist results.
- Run jobs support progress, cancellation, retry, duplicate, grouped results, and exports.
- Redacted backend panic/frontend error diagnostics are recorded under `.benchforge/diagnostics` and surfaced in Doctor.
- Latest full local release-readiness verification passed on 2026-07-09 with `make benchmark-readiness-full`, including clean first-run, provider contracts, Hugging Face GGUF search/download/start, local/cloud benchmarks, report export, unsigned DMG packaging, installed-app smoke, and local server cleanup. `make live-cloud-smoke` also skipped safely because no real provider keys were configured.

Not good enough yet:

- Benchmark packs are broad enough for early model selection and now support initial task weights plus uncertainty-discipline checks for higher-stakes comparisons, but still need calibrated public/private task suites before they should be treated as definitive.
- Local runtime management has reliable discovery contracts, but lifecycle management is still mostly llama.cpp-focused.
- Provider clients have stronger offline rate-limit and retry contracts now, but still need deeper live provider-specific streaming, retry, and cost coverage.
- Result analysis needs broader comparison workflows, but visible results now include metric coverage, an explicit comparison evidence grade, Wilson pass-rate intervals, task-target repetition warnings, exact pack/task-slot coverage warnings, and pricing-assumption warnings for cost evidence.
- Code/agent benchmarks have a safer worker bridge and basic result import support, but still need broader hardened imports for full benchmark-suite handoff.
- Clean-machine verification is repeatable through the full readiness and installed-DMG smoke gates, but distribution still needs signed/notarized release validation and more real-machine install coverage.

## Milestone 1: Reliable Target Setup

Goal: users can add local and cloud models without editing JSON or guessing what failed.

Deliverables:

- Complete target forms for local OpenAI-compatible, Ollama, LM Studio, llama.cpp, vLLM, MLX, OpenAI, Anthropic, Mistral, OpenRouter, Azure OpenAI, Google Gemini, and generic compatible APIs.
- Support target lifecycle cleanup from the UI. Status: edit, disable/enable, delete, and redacted-config copy flows are available for target rows while preserving historical runs and artifacts; disabled targets are visibly marked, skipped by run/comparison shortcuts, and must be revalidated after re-enabling; direct model rows also expose a **Compare** action when an enabled counterpart exists on the other side of the local/cloud split.
- Add authenticated model search where providers support it, with clear fallback to manual model entry. Status: live/preset catalog search covers OpenAI, Anthropic, Mistral, OpenRouter, Azure OpenAI resource model lists, and Google Gemini's OpenAI-compatible model list when the endpoint and key are configured; if a user pastes a cloud key before searching, the app stores it in Keychain before the catalog request.
- Validate targets with a tiny completion probe, not only `/models`. Status: initial implementation complete for OpenAI-compatible, OpenAI Responses, Anthropic, Mistral/OpenRouter, Azure OpenAI, and Google Gemini targets; adapter listing now performs fast level-2 checks for command availability, non-empty environment-key readiness, and default local `/models` endpoint reachability; target validation health is now persisted with status, detail, and checked timestamp, and edits clear stale health until the target is checked again; newly added direct model targets can also queue a tiny persisted, max-cost-capped `LLM Connectivity` sanity job after validation; `make validation-contract-smoke` now verifies stored cloud-style targets through `/models` plus completion probes against a loopback provider contract.
- Own first-run target handoffs in the backend. Status: new model target creation can now persist, validate, and queue the selected automatic benchmark in one backend command, including an existing local/cloud counterpart when the UI has one to compare against; the UI uses that path for new model targets, and `make create-target-handoff-smoke` verifies a catalog-selected OpenRouter-style target reaches validation, queued benchmark rows, positive cost metrics, provider-confirmed model identity, finish reason, HTTP metrics, report export, reproducibility metadata, and cloud classification.
- Classify validation failures as `missing_key`, `auth`, `model_not_found`, `endpoint_unreachable`, `timeout`, `unsupported_shape`, or `rate_limited`. Status: initial implementation complete with batch summaries grouped by code; validation probes now preserve HTTP status and structured provider error bodies, and `make validation-contract-smoke` verifies success, missing-key, auth, rate-limit, model-not-found, malformed-response, server-error, and unreachable-endpoint validation details.
- Keep all credentials in Keychain and redact them from logs, screenshots, target JSON, and exports. Status: Keychain-backed provider status also recognizes configured environment variables, the target form shows selected-adapter key readiness without exposing secret values, target creation rejects raw secret-shaped config keys, and cloud runs now fail fast when a required API key is missing.

Exit criteria:

- A fresh user can create one cloud target and one local target from the UI.
- Validation tells the user exactly whether the key, model name, endpoint, or runtime is wrong.

## Milestone 6: Safety and Sandboxing

Use `make benchmark-readiness` as the default local/cloud release gate. Use `BENCHFORGE_RELEASE_DISTRIBUTION=1 make package-release-dmg` plus `make verify-distribution-dmg` for a signed/notarized public DMG.

For sandbox work, also run:

```bash
make smoke-docker
```

`make smoke-docker` preflights the Docker CLI and daemon, requires Docker Desktop or Colima to be running, and currently applies network-off Docker scoring to Python repo/code tasks.

For real provider verification, set the relevant provider keys and run:

```bash
make live-cloud-smoke
BENCHFORGE_LIVE_CLOUD_RUN=1 BENCHFORGE_LIVE_CLOUD_PACK=llm-basics make live-cloud-smoke
```

This optional gate is intentionally separate from default readiness because it calls live APIs and may spend provider credit. Without `BENCHFORGE_LIVE_CLOUD_RUN=1`, it validates configured providers only.
